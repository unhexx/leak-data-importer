# Стандарты разработки

> Обязательные правила для всех участников цикла разработки.

## Язык и стиль

### Комментарии и документация
- **Все комментарии** в коде — на русском языке
- **DOCstrings** — на русском языке
- **Сообщения об ошибках** — на русском языке
- **README и документация** — на русском языке

Пример:
```python
def process_person_data(data: dict) -> Person:
    """Обрабатывает сырые данные персоны и создаёт объект Person."""
    # Проверяем обязательные поля
    if "name" not in data:
        raise ValueError("Отсутствует обязательное поле 'name'")
    
    return Person(name=data["name"])
```

### Git коммиты
- **Сообщения коммитов** — на русском языке
- **Стиль** — как реальный mid/senior разработчик

Примеры:
```
Добавлена валидация СНИЛС в модели персоны
Исправлена обработка пустого списка адресов
Добавлены тесты для экспорта в CSV
```

### Запрещённые слова
**НИКОГДА не используй** в коммитах и комментариях:
- AI, LLM, агент, нейросеть
- MiniMax, Grok, Claude, ChatGPT
- As an assistant, as an AI
- Generated, created by AI
- Bot, automated, machine learning

## Среда разработки

### Виртуальное окружение
- **Всегда используй .venv** — никогда не ставь пакеты в системный Python
- **Активация Windows**: `.venv\Scripts\activate`
- **Активация Unix**: `source .venv/bin/activate`

### Кодировка
- **Все файлы** — UTF-8
- **Не используй** cp1251 или другие кодировки

## Качество кода

### Форматирование
```bash
# Формат
.venv\Scripts\python -m ruff format .

# Линт
.venv\Scripts\python -m ruff check .
```

### Типизация
```bash
# Проверка типов
.venv\Scripts\python -m mypy src/
```

### Тесты
```bash
# Запуск тестов
.venv\Scripts\python -m pytest tests/ -v
```

## Безопасность

### PII (Персональные данные)
- **Никогда не логируй** реальные персоны
- **Используй PiiSafeLogger** для логирования
- **Тестовые данные** — только синтетические

### Учетные данные
- **Никогда не храни** пароли/ключи в коде
- **Используй** environment variables
- **Файл .gitignore** должен исключать credentials

## Deterministic принцип

### Основное правило
**Внутри pipeline никогда не вызывай LLM** — все преобразования должны быть детерминистичными.

### Допустимые использования LLM
- Планирование (Orchestrator)
- Написание кода (Coder)
- Отладка (Debugger)
- Ревью (Reviewer)

### Недопустимые использования LLM
- Внутр�� импорта данных
- Внутри нормализации
- Внутри экспорта
- Внутри построения графа
